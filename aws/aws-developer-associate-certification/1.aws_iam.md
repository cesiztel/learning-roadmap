## Introduction

IAM is the security backbone of everything in AWS. Before you can understand how any service is accessed or controlled, you need to understand these building blocks. Let me walk through each one.

## Building blocks

The core entities are what IAM is made of — the "nouns" of the system.

![alt text](iam_1.png)

* The **root account** is created when you first open an AWS account. It has unlimited power and cannot be restricted. The exam expects you to know: lock it away, use MFA on it, never use it for day-to-day work.
* An **IAM user** is a permanent identity with long-term credentials (username/password or access keys). Use it for human operators or legacy applications. The modern recommendation is to avoid access keys where possible.
* An **IAM group** is just a container for users — you attach policies to the group, and all members inherit them. Groups cannot contain other groups.
* An **IAM role** is the most important concept for developers. It has no permanent credentials — it issues temporary security tokens via STS. Services (Lambda, EC2), other accounts, or federated users assume roles. This is the preferred pattern for everything in AWS.

## Policies and policy evaluation

![alt text](iam_2.png)

The policy evaluation rule is one of the most exam-tested concepts: everything is denied by default (implicit deny), an explicit Allow grants access, but an explicit Deny anywhere in the chain always wins — no exceptions. Memorise this order.

## STS and Roles

Now the two concepts that trip up the most developers: STS and how roles are actually used in practice.

![alt text](iam_3.png)

STS (Security Token Service) is the engine behind roles. When a service or user calls AssumeRole, STS returns three things: an access key ID, a secret access key, and a session token — all temporary. This is the modern, correct way to grant permissions to anything in AWS, because the credentials expire automatically.

## Summary

* Entities to know cold: root account (dangerous, avoid), IAM users (humans/legacy apps), IAM groups (policy containers for users), IAM roles (preferred for services and cross-account access), IAM policies (JSON documents defining permissions).

* Policy mechanics: every statement has an Effect, Action, Resource, and optionally a Condition. The evaluation order is: explicit Deny → explicit Allow → implicit Deny (default).

* The role pattern: a role has two policies — a trust policy (who can assume it) and a permissions policy (what they can do after assuming it). This distinction catches a lot of people on the exam.

* Practical developer scenarios the exam loves: Lambda execution roles (Lambda needs a role to call other AWS services), EC2 instance profiles (attaches a role to an EC2 so it can call AWS APIs without hardcoded keys), cross-account access (role in Account B trusts Account A), and identity federation (users from an external IdP like Google or Active Directory assume a role).

* The golden rule: never hardcode AWS credentials in code. Always use roles and let the SDK pick up temporary credentials from the instance metadata or environment automatically.
